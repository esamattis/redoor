# REST API Implementation Plan for Connected Agents

## Overview
Add REST API endpoints for managing connected agents and executing commands (ls, cat) on them. Each endpoint will use dedicated response structs with ts-rs for TypeScript type generation.

## Implementation Approach

### 1. Add Response Structs with ts-rs (src/commands.rs)
Add new response structs after the existing CommandResult enum:
- `AgentListResponse` - List of connected agents
- `AgentInfoResponse` - Single agent info
- `LsResponse` - Directory listing result
- `CatResponse` - File content result
- `ErrorResponse` - Error response

All structs will use `#[ts(export)]` attribute for TypeScript generation.

### 2. Update RouterActor (src/actors/router.rs)
**New RouterMsg variants**:
- `GetAgentList { reply: RpcReplyPort<HashMap<String, String>> }` - Get list of connected agents via RPC
- `ExecuteCommandRest { agent_id: String, command: Command, reply: RpcReplyPort<CommandResult> }` - Execute command via RPC for REST API

**Handler updates**:
- Add handler for `GetAgentList`:
  - Extract agents from state.agents HashMap
  - Send back HashMap<String, String> (agent_id -> agent_name) via reply.send()
- Add handler for `ExecuteCommandRest`:
  - Store reply port in new field `rest_pending_responses: HashMap<String, RpcReplyPort<CommandResult>>` in RouterState (use agent_id as key, assuming one pending request per agent)
  - Route command to agent via existing RouteCommand with a placeholder originating_client
  - Modify RouteResponse handler to check both pending_responses (for WebSocket) and rest_pending_responses (for REST API)
  - Note: For concurrent REST requests to same agent, would need correlation_id (future enhancement)
- Import `ractor::RpcReplyPort` at top of file

### 3. Add REST Handler Functions (src/main.rs)
**Three new handler functions**:

1. `list_agents_handler`:
   - Use `ractor::call_t!` macro to call RouterMsg::GetAgentList with timeout
   - Convert HashMap to AgentListResponse with AgentInfoResponse items
   - Return JSON response or ErrorResponse on error

2. `ls_agent_handler`:
   - Extract agent_id from Path and directory path from TailPath
   - Use `ractor::call_t!` to call RouterMsg::ExecuteCommandRest with Ls command
   - Wait for CommandResult from agent
   - Return LsResponse or ErrorResponse on failure

3. `cat_agent_handler`:
   - Extract agent_id from Path and file path from TailPath
   - Use `ractor::call_t!` to call RouterMsg::ExecuteCommandRest with Cat command
   - Wait for CommandResult from agent
   - Return CatResponse or ErrorResponse on failure

All handlers should use a reasonable timeout (e.g., 30 seconds) and handle:
- Timeout errors â†’ HTTP 504
- Actor stopped errors â†’ HTTP 500
- CommandResult::Error â†’ HTTP 400/404 depending on message

### 4. Register Routes (src/main.rs)
Add routes to the Router builder:
- `/api/v1/agents` â†’ list_agents_handler
- `/api/v1/agents/:agent/ls/*path` â†’ ls_agent_handler
- `/api/v1/agents/:agent/cat/*path` â†’ cat_agent_handler

Need to import `Path`, `TailPath` from axum::extract.

### 5. Error Handling
| Error Case | HTTP Status | ErrorResponse |
|------------|-------------|---------------|
| Agent not found | 404 | error: "Agent not found" |
| Agent disconnected | 503 | error: "Agent disconnected" |
| Path not found | 404 | error from CommandResult::Error |
| Permission denied | 403 | error from CommandResult::Error |
| Timeout | 504 | error: "Request timeout" |
| Internal error | 500 | error: "Internal server error" |
