# Enhanced LS API Implementation Plan

## Objective
Add detailed file metadata to the ls API response including: file type, size, owner, group, UID, and GID.

## Files to Modify

1. **src/commands.rs** - Core implementation
   - Update `LsResult` and `LsResponse` structs
   - Add new `LsEntry` struct
   - Rewrite `ls()` function to collect metadata
   - Update tests

2. **Cargo.toml** - Add dependency for user/group lookups

3. **src/main.rs** - Update API handler (no changes needed, structure stays the same)

## Implementation Steps

### Step 2: Create new struct for file metadata

In `src/commands.rs`, add `LsEntry` struct after `LsResponse`:
```rust
#[derive(Debug, Clone, Serialize, Deserialize, TS)]
#[ts(export)]
pub struct LsEntry {
    pub name: String,
    #[serde(rename = "type")]
    pub file_type: String,
    pub size: u64,
    pub owner: Option<String>,
    pub group: Option<String>,
    pub uid: u32,
    pub gid: u32,
}
```

### Step 3: Update response structs

In `src/commands.rs`:

- Replace `LsResult { files: Vec<String> }` with `LsResult { files: Vec<LsEntry> }` (line 14)
- Replace `LsResponse { files: Vec<String> }` with `LsResponse { files: Vec<LsEntry> }` (line 62)

### Step 4: Rewrite ls() function

Replace the `ls()` function (lines 119-144) with implementation that:
- Uses `std::os::unix::fs::MetadataExt` for uid, gid, size
- Uses `nix::unistd::User::from_uid()` and `nix::unistd::Group::from_gid()` for owner/group lookups
- Returns `LsEntry` for each file instead of just filename string
- Handles errors gracefully with Option types

### Step 5: Update tests

Add assertions to verify metadata is present:
- Check `file_type` is "file" or "directory"
- Verify `uid` and `gid` are populated
- Verify structure contains all expected fields

### Step 6: Generate TypeScript bindings

Run: `./scripts/generate-ts-bindings`

This will generate `bindings/LsEntry.ts` and update `bindings/LsResponse.ts`.

### Step 7: Build and test

Add test to ui/tests/agents.test.ts

Run: `./scripts/build-and-test`
